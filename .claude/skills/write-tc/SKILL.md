# TC 작성

입력: $ARGUMENTS

---

## 0. 저장 규칙

### 파일 형식
- **JSON (.json)** 형식으로 로컬 저장
- Google Sheets 업로드 시 테이블 형식으로 변환

### 저장 경로
```
data/tc/{프로젝트}_{버전}_{기능명}_tc.json
```

**예시:**
- `data/tc/SAY_v1.2_프로세스_가이드_tc.json`
- `data/tc/BAY_v1.0_재고_관리_tc.json`

### 파일명 규칙
- 프로젝트명: 대문자 (SAY, BAY, SSO)
- 버전: `v{major}.{minor}` 형식
- 기능명: 한글 허용, 공백 대신 `_` 사용
- 접미사: `_tc.json`

### JSON 스키마

```json
{
  "metadata": {
    "product": "SAY",
    "version": "v1.2.0",
    "feature": "상담 프로세스 가이드",
    "created_at": "2026-01-26",
    "updated_at": "2026-01-26",
    "status": "초안",
    "source_scenario": "data/scenarios/SAY_v1.2_프로세스_가이드_scenario.md"
  },
  "test_cases": [
    {
      "id": "MON-1-001",
      "depth_1": "프로세스 가이드",
      "depth_2": "화면 구성",
      "depth_3": "상담 프로세스 유형",
      "depth_4": "초진 고객인 경우",
      "depth_5": "",
      "depth_6": "",
      "priority": "P2",
      "expected": "초진 프로세스(11단계) 자동 적용됨\n- 단계 순서: 첫인사, 아이스브레이킹, ...",
      "note": ""
    }
  ],
  "review_history": [
    {
      "round": 1,
      "type": "format|content",
      "result": "PASS|FAIL",
      "auto_fixed": ["위반 항목 자동 수정 내역"],
      "pending_approval": ["내용 수정 제안 (사용자 승인 필요)"],
      "timestamp": "2026-01-26T10:00:00"
    }
  ]
}
```

### TC ID 채번 규칙 (필수)

TC ID는 **화면명의 영문 약어 + 하위 메뉴 번호 + 순번**으로 구성한다.

**형식:**
| 구조 | 형식 | 예시 |
|------|------|------|
| 하위 메뉴 없음 | `{약어}-{순번}` | `CMN-001` |
| 하위 메뉴 있음 | `{약어}-{하위번호}-{순번}` | `MON-1-001`, `AI-3-001` |

**규칙:**
1. **약어**: 화면/기능명의 영문 약어 (대문자). 시나리오 또는 기획서의 메뉴 구조를 기반으로 결정
2. **하위 번호**: 같은 약어 내 하위 메뉴가 2개 이상이면 `-1`, `-2`, `-3`으로 구분. 하위 메뉴가 1개뿐이면 생략 가능
3. **순번**: 각 섹션 내에서 `-001`부터 시작, 3자리 zero-padding
4. **ID 고정 (필수)**: 한번 부여된 TC ID는 변경하지 않는다 (재채번 금지)
   - TC 삭제 시: 해당 ID는 빈 번호로 남겨둔다 (예: MON-1-003 삭제 → MON-1-002 다음 MON-1-004)
   - TC 추가 시: 해당 섹션의 마지막 순번 다음 번호를 부여한다 (예: MON-1-258 다음 → MON-1-259)
   - TC 이동(분산 통합) 시: 기존 ID를 유지한 채 위치만 이동한다
   - **이유**: JIRA, 버그 리포트, 리뷰 이력 등 외부에서 TC ID를 참조하므로 ID 변경 시 추적 불가

**예시 (SAY 프로젝트):**

| 화면/기능 | TC ID 범위 | 건수 |
|-----------|-----------|------|
| 공통 | CMN-001 ~ 080 | 80 |
| 상담 대시보드 | MON-1-001 ~ 258 | 258 |
| 상담 내역 | MON-2-001 ~ 059 | 59 |
| 상담 내역 상세 | MON-3-001 ~ 045 | 45 |
| 고객 관리 리스트 | CUST-1-001 ~ 105 | 105 |
| 고객 상세 | CUST-2-001 ~ 024 | 24 |
| AI 가이드 대시보드 | AI-1-001 ~ 154 | 154 |
| AI 가이드 관리 리스트 | AI-2-001 ~ 041 | 41 |
| AI 가이드 등록 | AI-3-001 ~ 089 | 89 |
| AI 가이드 상세 | AI-4-001 ~ 070 | 70 |
| AI 품질 점검 | AI-5-001 ~ 060 | 60 |

**약어 결정 방법:**
- TC 작성 시 시나리오의 1 Depth(대분류)와 2 Depth(화면명)를 기반으로 약어를 결정
- 신규 화면/기능이면 사용자에게 약어를 확인받은 후 채번
- 기존 TC가 있으면 기존 약어를 따름

### 컬럼 설명 (Google Sheets 템플릿 기준)

| Google Sheets 열 | 컬럼명 | JSON 키 | 용도 | 예시 |
|-----------------|--------|---------|------|------|
| A | (빈칸) | — | 가독성을 위해 항상 비워둠 | |
| B | TC ID | id | 테스트 케이스 고유 식별자 | MON-1-001, AI-3-001 |
| C | 1 Depth | depth_1 | 대분류/기능 | 제품 관리, SSO 로그인 화면 |
| D | 2 Depth | depth_2 | 화면/영역 | 검색 영역, 로그인 상태 |
| E | 3 Depth | depth_3 | UI 요소/테스트항목 | 구분 선택 드롭다운, [로그인] 버튼 |
| F | 4 Depth | depth_4 | 상태/조건 | Default, Tap, 미입력 상태 |
| G | 5 Depth | depth_5 | 세부 분기 | 다중 선택, 검색 결과 있는 경우 |
| H | 6 Depth | depth_6 | 추가 조건의 동작 | 토글 Tap, 문구 확인 |
| I | Priority | priority | 우선순위 | P1, P2, P3 |
| J | Expected Result | expected | 기대 결과 | 화면 진입됨, 리스트 노출됨 |
| K | 비고 | note | 비고 (선택) | 복수 지점 데이터 필요 |

> **A열 빈칸 규칙**: Google Sheets 업로드 시 A열은 항상 빈칸으로 둔다. TC 데이터는 B열(TC ID)부터 시작한다.

> **참조 템플릿:** `.claude/skills/write-tc/tc_template.csv`, `.claude/skills/write-tc/tc_template2.csv`
> **구조만 참고한다 (컬럼 순서, Depth 계층). CSV 내 TC 내용(테스트 항목, 기대결과 등)은 절대 가져오지 않는다.**
> **주의: 모든 행에 Depth 값을 빠짐없이 기입한다. 동일 값이라도 빈칸으로 두지 않는다.**
> TC 작성, 리뷰, Google Sheets 업로드 모두 이 컬럼 구조를 따른다.

3. **참조 시**: `/write-test-code`에서 TC JSON 읽어서 자동화 코드 생성

## 1. 기본 규칙

- **시나리오에 명시된 내용만 작성** (추측 금지)
- 기획서/정책서에 없는 내용 추가 금지 (Hover, 색상, 아이콘 등)
- 기존 TC 수정 시에도 시나리오 기준으로 검증 필수
- 모든 TC는 문서 출처를 명시할 수 있어야 함

### 데이터 소스 우선순위
1. **Confluence (기획서/정책서)** - 기본 기준
2. **Figma** - 문구 불일치 발견 시 참고
3. **로컬 JSON** - 참고용

### 문구 충돌 발견 시
- 기획자에게 동기화 요청
- TC에는 "(문구 확인 필요)" 표시
- 로컬 파일과 Confluence 충돌 시 → Confluence 기준

### QA 관점 추가 케이스
- TC에 직접 작성하지 않음
- TC 작성 완료 후 별도 "QA 검토 필요 항목" 목록으로 정리해서 사용자에게 전달

### 디자인 스펙 사용 금지 (필수)

| 유형 | 사용 금지 | 올바른 표기 |
|------|-----------|-------------|
| 너비 픽셀값 | w222, w58, w350 등 | (제거) 또는 "고정 너비 영역" |
| 색상 Hex 코드 | #5442FF, #FF4E51, #FBFBFB 등 | "파란색(Main Blue)", "빨간색(Main Red)" 등 색상명으로 단순화 |

**예시:**
```
❌ [저장] 버튼 (Filled, #5442FF, w222)
❌ 비활성화 상태 (Gray 500, #CBCBCB)
❌ 에러 문구 빨간색(#FF4E51)으로 노출됨

✅ [저장] 버튼이 활성화 상태로 노출됨
✅ 버튼이 비활성화 상태로 노출됨
```

## 4. TC 배치 및 Depth 구조 규칙

### 4.0 TC 배치 순서 (필수)

**핵심 원칙:**
1. **같은 상위 Depth를 한 블록으로 묶는다** — 같은 1 Depth(화면명) TC들은 반드시 연속 배치
2. **블록 내에서 확정된 시나리오 순서를 따른다** — 기획서에 명시된 순서, 없으면 논리적 흐름 순서 (진입 → 조회 → 등록 → 수정 → 삭제)
3. **Figma 보강 후 실제 화면 배치 순서가 확정되면 시나리오에서 반영** — TC는 확정된 시나리오 순서를 그대로 따름

> **참조:** `.claude/skills/write-tc/tc_template.csv`의 Depth 구조/행 순서만 참고하여 배치 (CSV 내 TC 내용은 가져오지 않음)
>
> **Figma export 활용 (배치 순서 검증):**
> - `figma_exports/{product}_{version}/` 폴더가 존재하면: description.txt + 이미지로 UI 배치 순서를 검증하여 TC 배치에 반영
> - 폴더가 없으면: 배치 순서 검증 SKIP → "배치 순서 미검증 (Figma export 없음)" 알림 출력 (TC 작성은 정상 진행)
> - **Figma 내용 비교(UI 요소/문구 대조)는 범위 밖** — 배치 순서 참고만 수행

TC는 확정된 시나리오의 순서를 따르되, 기본적으로 화면 UI 요소 배치 순서를 기준으로 한다:

```
1 Depth: 대분류/기능 (메뉴 대분류)
  2 Depth: 화면명 (메뉴 진입 경로)
    3 Depth: 화면 영역 (상→하, 좌→우 순서)
      4 Depth: UI 요소 (해당 영역 내 배치 순서)
```

**화면 영역 순서:**
```
① 간편 메뉴 (최상단 헤더) → ② 검색 영역 → ③ 버튼 영역 (리스트 상단)
→ ④ 리스트 영역 (정렬, 페이지네이션) → ⑤ 등록/수정 화면 → ⑥ Footer
```

**메뉴 배치 순서 예시 (참고 TC 기반):**
```
제품
 ├── 제품 관리
 │    ├── 간편 메뉴 → [제품] 선택 → [제품 관리] 선택
 │    ├── 검색 영역 → 구분 드롭다운 → 종류 드롭다운 → 제조사 드롭다운 → 제품명 검색
 │    ├── [엑셀 다운로드] / [제품 등록] 버튼
 │    ├── 리스트 영역 → 정렬 → 페이지네이션 → [일괄수정] → [수정] → [일괄삭제] → [삭제]
 │    ├── 제품 등록 → 뒤로가기 → 입력 영역 → [저장] 버튼
 │    └── 제품 수정 → 뒤로가기 → 수정 영역 → [저장] / [삭제] 버튼
 ├── 카테고리 관리 (메뉴 순서 2)
 └── 발주처 관리 (메뉴 순서 3)
```

### 4.0.1 예외/에러 케이스 배치 규칙 (필수)

**핵심 원칙: 예외 케이스를 별도 섹션으로 분리하지 않는다. 해당 UI 요소의 하위 조건으로 인라인 배치한다.**

```
✅ 올바른 배치 (인라인):
[일괄삭제] 버튼
  ├── 선택된 체크박스가 없는 경우 → 토스트 "삭제할 제품을 선택해주세요"
  ├── 선택된 체크박스가 있는 경우
  │    ├── 1개인 경우 → Alert "제품을 삭제하시겠습니까?"
  │    │    ├── [취소] 버튼 → 화면 유지
  │    │    └── [삭제] 버튼 → 삭제 완료 토스트
  │    └── n개인 경우 → Alert "n개의 제품을 삭제하시겠습니까?"
  ├── 현재 발주 중인 제품 포함 → 토스트 "발주 중인 제품은 삭제할 수 없습니다"
  └── 재고 남아있는 제품 포함 → 토스트 "재고가 남아 있어 삭제할 수 없습니다"

❌ 잘못된 배치 (분리):
[일괄삭제] 버튼 → 정상 동작만 작성
...
예외 케이스 (별도 섹션)         ← 분리 금지
  ├── 발주 중인 제품 삭제 시
  └── 재고 남은 제품 삭제 시
```

**규칙:**
- 에러 토스트, 삭제 불가, 권한 없음 등 예외는 해당 동작의 **조건 분기**로 배치
- 같은 버튼의 정상/예외가 하나의 Depth 트리 안에서 표현
- 뒤로가기, 변경사항 미저장 Alert 등도 해당 화면의 UI 요소 하위에 배치

### Depth 컬럼별 역할

- **4 Depth (테스트항목)**: UI 요소명 (예: [AI 가이드 등록] 버튼, 상태 드롭다운)
- **5 Depth (세부케이스)**: 상태/조건 (예: Default, Tap)
- **6 Depth (추가조건)**: 세부 분기 (예: 활성화 선택, 비활성화 선택)
- **7 Depth (추가조건의 동작/결과)**: 6 Depth 조건에서의 동작 (예: 토글 Tap, 문구 확인)

**순서 규칙:**
- 없는 경우 → 있는 경우 순서로 작성
- Default → Tap 순서 (Hover는 시나리오에 명시된 경우만)

### 4.0.2 Depth 중복 방지 규칙 (필수)

**핵심 원칙: 인접 Depth 간 동일/유사 의미 반복 금지**

시나리오의 섹션 제목(#, ##, ###)을 Depth에 그대로 매핑하지 않는다.
의미가 겹치면 하나로 합치고, 하위 내용을 한 단계씩 올린다.

**판단 기준:**
- depth_N과 depth_N+1이 동일 단어를 포함하면 중복
- depth_N이 depth_N+1의 상위 개념이면서 depth_N+1 외 다른 하위가 없으면 중복

**예시:**
| 중복 (금지) | 수정 |
|-------------|------|
| depth_2: 비정상 종료 처리 / depth_3: 비정상 종료 시 처리 | depth_2: 비정상 종료 처리 / depth_3: (하위 항목으로 바로 이동) |
| depth_2: 예외 처리 / depth_3: 네트워크 오류 / depth_4: 네트워크 연결 실패 | depth_2: 예외 처리 / depth_3: 네트워크 연결 실패 |
| depth_3: 상품 목록 API 실패 / depth_4: 상품 목록 조회 실패 | depth_3: 상품 목록 조회 실패 |
| depth_3: 결제 정보 저장 실패 / depth_4: [결제 정보 저장] 클릭 후 저장 실패 | depth_3: 결제 정보 저장 실패 |

**적용 시점:** TC 작성 완료 후, 전체 TC를 스캔하여 인접 depth 간 중복을 검출하고 자동 수정한다.

### 4.1 계층 구조 원칙 (중요)

**핵심 원칙: 조건 하위에 해당 동작/결과 배치**
- 특정 조건에서만 발생하는 동작은 해당 조건의 하위 Depth에 배치
- 동일한 동작이라도 조건에 따라 결과가 다르면 각 조건 하위에 별도 배치
- 모든 조건에 공통인 동작은 상위 레벨에 배치

**패턴 1: 검색 결과 있음/없음**
```
5 Depth: 텍스트 입력
  6 Depth: 검색 결과 있는 경우
    7 Depth: 검색어 변경/삭제 시
  6 Depth: 검색 결과 없는 경우
    7 Depth: 문구 확인
```

**패턴 2: 버튼 활성화 조건**
```
4 Depth: [초기화] 버튼
5 Depth: 검색 필드 미입력 상태
5 Depth: 검색 필드 입력 상태
  6 Depth: 버튼 Tap
```

**패턴 3: 토글 상태별 동작**
```
4 Depth: 활성화 토글
5 Depth: 토글 상태
  6 Depth: 활성화 상태인 경우
    7 Depth: 토글 Tap → 비활성화로 변경
  6 Depth: 비활성화 상태인 경우
    7 Depth: 토글 Tap → 활성화로 변경
5 Depth: 토글 변경 반영
5 Depth: 진행 중 상담 영향
```

**패턴 4: 메뉴 항목 동작**
```
4 Depth: 더보기 [⋮] 메뉴
5 Depth: Default
5 Depth: [⋮] 버튼 Tap
5 Depth: [상세] 메뉴
  6 Depth: Hover
  6 Depth: Tap
```

**패턴 5: 버튼 활성화/비활성화 조건 분리**
```
4 Depth: [완료] 버튼
5 Depth: 비활성화 조건
  6 Depth: 가이드명 미입력
  6 Depth: 가이드 내용 미입력
5 Depth: 활성화 조건
  6 Depth: 가이드명 AND 가이드 내용 입력됨
5 Depth: 버튼 Tap
  6 Depth: 성공 시
  6 Depth: 실패 시
```

**패턴 6: Alert 내 버튼 동작**
```
4 Depth: [취소] 버튼
5 Depth: 입력 있음
  6 Depth: Alert 노출
    7 Depth: [취소] Tap
    7 Depth: [확인] Tap
5 Depth: 입력 없음
```

**잘못된 예시 (평면적 구조):**
```
❌ 5 Depth: 검색 결과 있는 경우
❌ 5 Depth: 검색 결과 없는 경우
❌ 5 Depth: "검색 결과가 없습니다" 문구 확인  ← 이건 "없는 경우" 하위여야 함
❌ 5 Depth: 리스트 항목 클릭                 ← 이건 "있는 경우" 하위여야 함
```

## 5. Expected Result 작성 규칙

**핵심 원칙:**
- 기대결과에는 **결과만 작성** (조건/분기는 Depth에서 표현)
- 하나의 기대결과에는 **하나의 결과만** 작성
- 부가 설명이 필요한 경우 줄바꿈 후 `- ` 사용

**문체 규칙:**
- **긍정**: ~됨, ~적용됨, ~표시됨, ~노출됨, ~활성화됨
- **부정**: ~되지 않음, ~불가함, ~없음
- **지양**: ~된다, ~합니다, ~해야 함 (서술형 혼용 금지)

**금지:**
- 기대결과에 "~한 경우", "~일 때" 등 조건 포함
- 하나의 Expected Result에 "노출됨" 2회 이상 사용
- 쉼표(,)로 여러 결과 연결
- 시나리오에 없는 내용 추가

**권장:**
```
❌ "전체 선택 상태로 노출됨, 모든 가이드 표시됨"
✅ "전체 선택 상태로 노출됨
   - 모든 가이드 표시됨"

❌ "드롭다운 목록 노출됨\n- 전체 / 활성화 / 비활성화 항목 노출됨"
✅ "드롭다운 목록 노출됨 (전체 / 활성화 / 비활성화)"
```

**버튼 활성화/비활성화 TC 올바른 구조:**

✅ 올바른 예시:
| 4 Depth | 5 Depth | 6 Depth | Priority | Expected Result |
|---------|---------|---------|----------|-----------------|
| [AI로 가이드 생성] 버튼 | 모든 입력 비어있음 | | P3 | 버튼 비활성화됨 |
| [AI로 가이드 생성] 버튼 | 하나 이상 입력됨 | AI 텍스트만 입력됨 | P2 | 버튼 활성화됨 |
| [AI로 가이드 생성] 버튼 | 하나 이상 입력됨 | 파일만 업로드됨 | P2 | 버튼 활성화됨 |

### 시나리오 변경사항 반영 (TC 수정 모드)

**기본 규칙:**
- 삭제된 기능: TC에서 완전 제거
- 기본값 변경: TC 즉시 반영

**TC 수정 시 뎁스 구조 규칙 (필수):**

시나리오 변경으로 기존 TC를 수정할 때, 반드시 아래 절차를 따른다.

1. **기존 뎁스 구조 탐색**: 수정 전 기존 TC의 전체 뎁스 트리를 먼저 파악한다 (특히 depth_1~3의 메뉴/화면명 목록을 먼저 수집)
2. **삽입 위치 판단**: 변경/추가 내용이 기존 트리의 어디에 해당하는지 찾아서 해당 위치에 수정/삽입한다
3. **동일 뎁스 값 중복 금지**: 같은 레벨에 이미 존재하는 뎁스 값이면 해당 노드에 합류한다 (신규 노드 생성 금지)
4. **수정 후 중복 스캔 (필수)**: 수정 완료 후 **전체 TC를 depth_1 → depth_2 → depth_3 순서로 스캔**하여 동일/유사 값 중복을 검출하고 병합한다

**⚠️ 특히 주의: Depth 2-3 메뉴/화면명 중복**

TC 수정 시 가장 빈번하게 발생하는 오류는 **depth_2(화면명), depth_3(영역명)이 동일한 값으로 2~3개씩 생성**되는 것이다.

**수정 후 중복 스캔 절차:**
1. depth_1 기준으로 TC를 그룹핑한다
2. 각 그룹 내에서 depth_2 값의 고유 목록을 추출한다
3. 동일 depth_2 값이 2개 이상 비연속 블록으로 존재하면 → 첫 블록에 병합
4. 병합된 블록 내에서 depth_3도 동일하게 검사 → 중복 시 병합
5. 병합 시 기존 TC ID 유지 (위치만 이동, 재채번 금지)

**예시 1 - depth_3 중복 (가장 빈번):**
```
❌ 수정 후 중복 발생:
2 Depth: 제품 관리
  3 Depth: 검색 영역          ← TC-005~010
    4 Depth: 구분 드롭다운
    4 Depth: 제품명 검색
  3 Depth: 리스트 영역         ← TC-011~020
    4 Depth: ...
  3 Depth: 검색 영역           ← TC-030~035 (수정 시 새로 생성됨 — 금지!)
    4 Depth: 카테고리 필터

✅ 올바른 수정 (기존 노드에 병합):
2 Depth: 제품 관리
  3 Depth: 검색 영역           ← TC-005~012 (카테고리 필터가 기존 블록에 삽입됨)
    4 Depth: 구분 드롭다운
    4 Depth: 카테고리 필터      ← 여기에 삽입
    4 Depth: 제품명 검색
  3 Depth: 리스트 영역          ← TC-013~022
    4 Depth: ...
```

**예시 2 - depth_2 중복:**
```
❌ 수정 후 중복 발생:
1 Depth: 제품
  2 Depth: 제품 관리           ← TC-001~050
  2 Depth: 카테고리 관리        ← TC-051~080
  2 Depth: 제품 관리           ← TC-081~090 (금지!)

✅ 올바른 수정:
1 Depth: 제품
  2 Depth: 제품 관리           ← TC-001~060 (TC-081~090이 여기로 병합)
  2 Depth: 카테고리 관리        ← TC-061~080
```

### Depth 분산 자동 통합 (필수)

TC 작성/수정 완료 후, **depth_1 ~ depth_3까지 동일 값의 TC가 비연속적으로 흩어져 있는지 전체 스캔**한다.

**스캔 순서 (필수):**
1. **depth_1 스캔**: 동일 depth_1 값이 비연속 분산되어 있으면 → 첫 블록에 병합
2. **depth_2 스캔**: 같은 depth_1 내에서 동일 depth_2 값이 비연속 분산되어 있으면 → 첫 블록에 병합
3. **depth_3 스캔**: 같은 depth_2 내에서 동일 depth_3 값이 비연속 분산되어 있으면 → 첫 블록에 병합
4. 병합 시 기존 TC ID 유지 (위치만 이동, 재채번 금지)
5. 분산 통합은 양식 수정이므로 사용자 승인 없이 자동 수행

> **TC 수정 시 특히 주의**: 시나리오 변경을 반영하면서 depth_2(화면명), depth_3(영역명)이 동일 값으로 2~3개씩 생성되는 경우가 빈번함. 반드시 스캔 수행할 것.

## 9. 테스트 기법별 커버리지 검토

TC 작성 완료 후, 아래 테스트 기법이 적절히 반영되었는지 리뷰한다.
**시나리오에 근거가 있는 항목만 TC로 추가하며, 근거 없이 추측으로 추가하지 않는다.**

### 테스트 기법 체크리스트

| 기법 | 설명 | 확인 포인트 | 예시 |
|------|------|-------------|------|
| **경계값 테스트** | 경계 근처 값 검증 | 입력 필드 최대/최소, 리스트 최대 표시 개수 | 30자/31자 입력, 최대 5개 표시 |
| **동등 분할** | 유효/무효 그룹 대표값 | 정상/비정상 입력 그룹 분리 | 정상 이메일 / 형식 오류 이메일 |
| **조건 분기** | 조건별 다른 결과 | 개수/상태에 따른 동작 차이 | 0개→미노출, 1개→노출, 2개 이상→통합 노출 |
| **상태 전이** | 상태 변화 흐름 검증 | 토글, 상태 변경, 모드 전환 | 활성→비활성, 조회→수정 모드 |
| **에러 추측** | 경험 기반 오류 예측 | 네트워크 끊김, 중복 데이터, 동시 접근 | 저장 실패, 가이드명 중복 |
| **긍정/부정 테스트** | 정상/비정상 동작 쌍 | 성공/실패 케이스 쌍으로 존재하는지 | 등록 성공 / 등록 실패 |
| **권한 테스트** | 권한별 접근/동작 차이 | 관리자/일반 사용자 분리, URL 직접 접근 | 관리자→메뉴 노출, 직원→미노출 |

### 리뷰 절차

TC 작성 완료 후 아래 순서로 리뷰를 수행한다:

1. **기법별 스캔**: 위 체크리스트의 각 기법을 시나리오에 대입하여 누락 여부 확인
2. **누락 판단**: 시나리오/기획서/정책서에 근거가 있는데 TC에 빠진 항목 식별
3. **추가 제안**: 누락된 TC를 사용자에게 목록으로 제안 (자동 추가하지 않음)
4. **승인 후 추가**: 사용자 승인 시 TC에 반영

### 리뷰 출력 형식

```
## 테스트 기법별 커버리지 리뷰

### 충족됨
- [x] 경계값 테스트: 글자수 제한 (30자/31자), 최대 5개 표시
- [x] 상태 전이: 토글 ON/OFF, 조회→수정 모드 전환

### 추가 제안
| # | 기법 | 위치 | 추가 TC 내용 | 근거 |
|---|------|------|-------------|------|
| 1 | 조건 분기 | 점검 필요 항목 | 0개/2개 이상 동작 분기 | 정책서 4-1 |
| 2 | 에러 추측 | 가이드 등록 | 동시 수정 충돌 | 정책서 6-3 |

### 해당 없음
- 동등 분할: 입력 필드 없음 (대시보드 화면)
```

## 11. 자체 리뷰 루프 (필수)

TC 작성/수정 완료 후, 사용자에게 보고하기 전에 반드시 **최소 2회 자체 리뷰**를 수행한다.

### 리뷰 루프 절차

```
TC 작성/수정 완료
  → 1차 리뷰 (check-format + review-tc + review-qa)
    → 양식 위반 자동 수정
    → 내용 수정 제안 목록 작성
  → 2차 리뷰 (check-format + review-tc + review-qa)
    → 양식 위반 자동 수정
    → 내용 수정 제안 목록 보완
  → 2차까지 양식 PASS → 사용자에게 최종 보고
```

### 자동 수정 vs 승인 필요 구분

| 구분 | 범위 | 처리 |
|------|------|------|
| **양식(format) 수정** | 용어 교정, 표기 통일, Depth 구조 정리, 분산 통합 (ID 유지) | **자동 수정** (사용자 보고 불필요) |
| **내용(content) 수정** | TC 추가/삭제, 우선순위 변경, Expected Result 변경 | **사용자 보고 후 승인받고 반영** |

### 반복 조건

- 2차 리뷰에서도 양식 피드백이 있으면 → 자동 수정 후 재리뷰 (PASS까지 반복, **최대 5회**)
- 중간 양식 수정 과정은 사용자에게 보고하지 않음
- 5회 초과 시 사용자에게 에스컬레이션

### 최종 보고 시 포함 항목

1. **양식 수정 이력**: 자동으로 수정한 항목 요약 (수정 전 → 후)
2. **내용 수정 제안 목록**: 사용자 승인이 필요한 변경 사항
3. **리뷰 결과 요약**: 커버리지, Pass/Fail 판정

### review_history 기록

리뷰 수행 시마다 JSON의 `review_history` 배열에 기록:
```json
{
  "round": 1,
  "type": "format",
  "result": "FAIL",
  "auto_fixed": ["Rule 9: TC-015 depth_2 빈칸 → '검색 영역' 채움", "Rule 5: TC-022 '테이블' → '리스트' 교정"],
  "pending_approval": [],
  "timestamp": "2026-01-26T10:00:00"
}
```

## 12. 출력 형식

위 **0. 저장 규칙**의 JSON 스키마 참조.

### 간략 형식 (단일 TC)

```json
{
  "id": "AI-1-001",
  "depth_1": "대분류/기능",
  "depth_2": "화면/영역",
  "depth_3": "UI 요소",
  "depth_4": "상태/조건",
  "depth_5": "세부 분기",
  "depth_6": "추가 조건",
  "priority": "P1/P2/P3",
  "expected": "기대 결과",
  "note": "비고 (선택)"
}
```

> **주의: 모든 행에 Depth 값을 빠짐없이 기입한다. 동일 값이라도 빈칸으로 두지 않는다.**

### Google Sheets 업로드

TC JSON을 Google Sheets에 업로드할 때는 테이블 형식으로 변환한다. **A열은 항상 빈칸**, TC 데이터는 B열부터 시작:

| (A: 빈칸) | TC ID | 1 Depth | 2 Depth | 3 Depth | 4 Depth | 5 Depth | 6 Depth | Priority | Expected Result | 비고 |
|-----------|-------|---------|---------|---------|---------|---------|---------|----------|-----------------|------|
| | {id} | {depth_1} | {depth_2} | {depth_3} | {depth_4} | {depth_5} | {depth_6} | {priority} | {expected} | {note} |

### Google Sheets 업로드 시 양식 참조

Google Sheets에 TC를 업로드할 때, 아래 엑셀 템플릿의 **구조와 양식**을 참조하여 시트를 구성한다.

- **참조 파일:** `.claude/skills/write-tc/tc_template.csv`, `.claude/skills/write-tc/tc_template2.csv`
- **참조 범위:** 컬럼 구조, 헤더 명칭, 셀 병합 방식, 시트 레이아웃, 표현 형식
- **참조하지 않는 것:** TC 내용 (내용은 반드시 JSON 데이터 기준으로 작성)

#### 참조 항목

| 항목 | 설명 |
|------|------|
| 컬럼 순서/명칭 | 엑셀 템플릿의 헤더 순서와 명칭을 따름 |
| Depth 값 기입 | 모든 행에 Depth 값을 빠짐없이 채움 (셀 병합/빈칸 처리 사용하지 않음) |
| 시트 레이아웃 | 헤더 행 위치, 데이터 시작 행, 고정 행/열 등 |
| 표현 형식 | 우선순위 표기, 기대결과 줄바꿈, 비고란 활용 등 |
