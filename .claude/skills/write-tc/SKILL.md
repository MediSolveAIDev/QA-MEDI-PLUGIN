# TC 작성

입력: $ARGUMENTS

> **본 스킬은 `.claude/rules/tc_writing_rule.md` 규칙을 따른다. 충돌 시 규칙 문서 우선.**
> 분류 기준·사전조건 누적·분해 알고리즘·약어 ID 형식 등 상세 룰은 규칙 문서 참조.

---

## 0. 학습 규칙 사전 반영 (TC 작성 전 필수)

TC 작성 전 `data/rules/` 폴더에서 **active 상태의 규칙**을 로드하여 TC에 사전 반영한다.

### 로드 절차

1. `data/rules/` 폴더 스캔
2. 각 규칙 JSON에서 `status: "active"` + `auto_apply: true` 필터
3. `projects`에 현재 프로젝트가 포함된 규칙만 선택
4. 시나리오 내용과 규칙의 `pattern`을 대조하여 해당하는 규칙 식별

### 적용 방식

| 조건 | 액션 |
|------|------|
| 규칙의 패턴이 현재 시나리오의 UI/기능과 일치 | 해당 규칙의 `recommendation`을 TC로 사전 작성 |
| 규칙의 패턴이 현재 시나리오와 무관 | 스킵 |
| `auto_apply: false`인 규칙 | TC에 자동 반영하지 않음 (리뷰 에이전트가 체크) |

> 규칙이 없으면 (폴더 비어있음) 이 단계를 건너뛰고 시나리오 기반으로만 TC를 작성한다.

## 1. 저장 규칙

### 파일 형식
- **JSON (.json)** 형식으로 로컬 저장
- Google Sheets 업로드 시 테이블 형식으로 변환

### 저장 경로
```
data/tc/{프로젝트}_{버전}_{기능명}_tc.json
```

### 파일명 규칙
- 프로젝트명: 대문자 (예: SAY, BAY, DAY)
- 버전: `v{major}.{minor}.{patch}` 형식
- 기능명: 한글 허용, 공백 대신 `_` 사용
- 접미사: `_tc.json`

### JSON 스키마 (필수)

```json
{
  "metadata": {
    "product": "DAY",
    "version": "v0.1.0",
    "feature": "예약관리",
    "abbreviation": "RSV",
    "created_at": "2026-05-12",
    "updated_at": "2026-05-12",
    "status": "초안",
    "source_scenario": "data/scenarios/DAY_v0.1.0_예약관리_scenario.md"
  },
  "test_cases": [
    {
      "id": "RSV-FLT-001",
      "category_l1": "예약관리",
      "category_l2": "좌측 필터 패널",
      "category_l3": "필터 패널 토글",
      "precondition": "1. 좌측 필터 패널 펼쳐진 상태",
      "step": "1. 좌측 필터 패널 접기 버튼 Tap",
      "expected": "좌측 필터 패널이 접힘\n   - 메인 콘텐츠 영역이 좌측까지 확장됨",
      "priority": "P3",
      "note": ""
    }
  ],
  "review_history": []
}
```

### 컬럼 설명 (Google Sheets 기준)

| 시트 컬럼 | JSON 필드 | 용도 |
|---|---|---|
| A | (빈칸) | 가독성을 위해 항상 비워둠 |
| B | id | TCID (`<도메인>-<L2약어>-<NNN>`) |
| C | category_l1 | 대분류 — 화면 (헤더 메뉴 단위) |
| D | category_l2 | 중분류 — UI 영역 (기획서 § 단위 우선) |
| E | category_l3 | 소분류 — UI 요소·동작 |
| F | precondition | 사전조건 (`1. /2. /3.` 번호 매김) |
| G | step | Step (`1. /2. /3.` 번호 매김) |
| H | expected | 기대결과 (번호 없음, 서브 불릿 허용) |
| I | priority | P1 / P2 / P3 |
| J~M | Test Result (Result/담당자/Date/BTS ID) | 실행 결과 |
| N | note | 액션 필요 항목만 (M/G/C 코드·자유 텍스트) |

## 2. TC ID 채번 규칙

**형식**: `<도메인약어>-<L2약어>-<NNN>` (하이픈 구분 3단)

- 예: `RSV-FLT-001`, `PAY-FRM-005`, `CMN-LGN-012`
- 도메인 약어 (3자) / L2 약어 (3자) / 시퀀스 (3자리 zero-padded)

**규칙**:
- 같은 (도메인, L2) 안에서 001부터 순차 부여
- 분해된 각 행은 **별개의 unique TCID** (sub-index `_01`, `_02` 사용 X)
- 분류 변경 시 L2 약어 바뀌면 새 ID로 갈음
- 삭제 시 번호 재사용 X (이력 보존)

**약어 사전 작성**:
- 프로젝트 시작 시 도메인별 L2 약어 사전 정의
- 신규 L2 등장 시 약어 사전 갱신
- 가상 L2 약어 권장: `ALL` (전체) / `TBD` (미구현) / `RTM` (실시간 동기화) / `PRM` (권한 검증) / `STA` (상태 전이)

> 약어 결정이 모호하면 사용자에게 확인 후 채번.

## 3. 분류 기준 (L1 / L2 / L3)

> 상세는 `.claude/rules/tc_writing_rule.md` §1 참조.

**패턴 A (화면 중심) 기본 권장**

| 레벨 | 기준 |
|---|---|
| L1 (대분류) | 화면 단위 — 헤더 메뉴명 그대로 |
| L2 (중분류) | UI 영역 — 기획서 § 섹션 단위 우선 |
| L3 (소분류) | UI 요소·동작 — 영역 내 개별 위젯·액션 |

**가상 L2 (Cross-cutting·상태 케이스)**
- Cross-cutting: `실시간 동기화` / `권한 검증` / `상태 전이`
- 상태: `(전체)` / `(미구현)`

**원칙**: 분류는 "어디서"만 표현. 동사·조건·케이스 유형은 사전조건·note로.

---

## 4. 사전조건 (Given)

> 상세는 `.claude/rules/tc_writing_rule.md` §2 참조.

**형식**: `N. ` 번호 매김 필수 (1줄이어도 `1. ` prefix)

**로그인·권한 처리 (절충 방침)**:
- 일반 TC: 로그인 상태 명시 불필요 (기본 전제)
- 화면 진입 검증·로그인 테스트: 명시 필요
- 동작 권한 필요·권한 분기: 명시 필요
- 단순 조회·UI 검증: 권한 정보 제거

**분해 행 사전조건 누적 (옵션 A)**:
- 후속 행이 직전 결과를 데이터·도메인 상태로 전제하면 누적
- 일시 UI (창/팝업/드롭다운/모달/토스트) → 누적 X (Step에 자체 표현)
- 영구 변경 (등록됨/변경됨/삭제됨 등) → 누적 O
- 모호 → 누적 X (보수적)

---

## 5. Step (When)

> 상세는 `.claude/rules/tc_writing_rule.md` §3 참조.

**형식**: `N. ` 번호 매김 필수 (1줄이어도 `1. ` prefix)

**규칙**:
- 한 Step = 한 사용자 액션
- Step과 Expected는 행 단위 1:1 매핑 — 멀티 step은 행 분해
- Step 텍스트로 자기 충족성 확보 (예: "X 창에서 [확인] Tap")

---

## 6. 기대결과 (Then) + 분해 알고리즘

> 상세는 `.claude/rules/tc_writing_rule.md` §4 참조.

**형식**: 번호 매김 없이 한 행에 한 결과만. 서브 불릿(`   - ...`)으로 세부 검증 묶음.

**분해 알고리즘 (Step 그룹 단위)**:

1. Step과 Expected 수 매칭:
   - Step N == Expected N → 1:1 매핑
   - Step 1 + Expected N → 한 TC, expected 서브 불릿
   - Step N + Expected 1 → 한 TC, 모든 step을 한 셀에 묶음 (번호 재부여)
   - Step M + Expected N (M<N) → 처음 M-1은 1:1, 나머지는 마지막 step에 묶임
2. 인접 같은 step 인덱스 = 한 그룹 (= 한 TC 행)
3. 다른 step 인덱스 = 별도 그룹 (= 별도 TC 행, 연쇄)
4. 한 그룹 안 멀티 expected → 첫 expected(메인) + 나머지 서브 불릿
5. **분해된 각 행의 Step 번호는 1부터 재부여** (원본 번호 그대로 X — 한 행 안에서 1, 2, 3...)

**별도 부모 TC 분리 케이스 (분해 X)**:
- 동일 시작점, 다른 액션 조합 → 별도 부모 TC
- 동일 액션, 다른 사전조건/데이터 → 별도 부모 TC
- 동일 액션, 다른 권한 결과 분기 → 별도 부모 TC

---

## 7. Note (비고)

> 상세는 `.claude/rules/tc_writing_rule.md` §6 참조.

**액션 필요 항목만 기재** — 정상 케이스는 빈칸.

- `M` 코드: 미구현
- `G` 코드: 기획 부재
- `C` 코드: 정책 충돌
- 자유 텍스트: "기획 확인 필요" 등

**비고에 넣지 않는 것**: 케이스 유형 분류, 일반 정책 인용, 이슈 키(`{PROJECT}-XX`)

## 8. Priority 기준

| 등급 | 적용 케이스 |
|---|---|
| P1 | 핵심 흐름 (smoke). 권한·인증·결제·핵심 상태 전이 |
| P2 | 주요 기능. 경계·네거티브 중 영향 큰 것. 실시간. 정상 흐름 변형 |
| P3 | 보조 기능·UI 디테일. 미구현 항목 |

**기본 배분 가이드**: P1 ~25% / P2 ~45% / P3 ~30%

## 9. 기본 작성 원칙

- **시나리오에 명시된 내용만 작성** (추측 금지)
- 기획서/정책서에 없는 내용 추가 금지 (Hover, 색상, 아이콘 등)
- 기존 TC 수정 시에도 시나리오 기준으로 검증 필수
- 본문(precondition·step·expected)에는 **비즈니스 용어만** — 개발 용어(DB 컬럼명·상태 코드·HTTP 코드)는 note 컬럼에만

### 데이터 소스 우선순위
1. **Confluence (기획서/정책서)** — 기본 기준
2. **Figma** — 문구 불일치 발견 시 참고
3. **로컬 JSON** — 참고용

### 문구 충돌 발견 시
- 기획자에게 동기화 요청
- 비고에 "기획 확인 필요" 표시
- 로컬 파일과 Confluence 충돌 시 → Confluence 기준

## 10. 필수 포함 TC

| 유형 | 필수 케이스 |

## 11. 용어 규칙

| 사용 금지 | 대체 용어 |
|---|---|
| 테이블 | 리스트 |
| 컬럼 | (제거) 또는 "영역" |
| 버튼 텍스트 | (제거) — 버튼명만 표기 |
| 헤더 | 리스트 헤더 또는 제거 |

### 디자인 스펙 사용 금지 (필수)

**TC에는 디자인 구현 상세(픽셀, 색상 코드, 디자인 토큰, 스타일 스펙)를 포함하지 않는다.**

| 유형 | 사용 금지 | 올바른 표기 |
|------|-----------|-------------|
| 너비 픽셀값 | w222, w58, w350 등 | (제거) 또는 "고정 너비 영역" |
| 색상 Hex 코드 | #5442FF, #FF4E51 등 | "파란색", "빨간색" 등 색상명 |
| 디자인 토큰명 | Gray 500, Main Blue 등 | "비활성화 상태", "활성화 상태" 등 기능적 상태 |
| 버튼 스타일 스펙 | 아웃라인, Filled, 위치/스타일 | (제거) — 버튼명과 동작만 |

### 활성화/비활성화 표기 규칙

영문 용어(Active, Disable, Disabled, Enable 등) 사용 금지. 한국어 서술형으로:
- ✓ `[저장] 버튼이 활성화 상태로 노출됨`
- ✓ `[저장] 버튼이 비활성화 상태로 노출됨`
- ✗ `Disabled` / `Active 상태`

### 상태 표기

| 항목 | 표기 방식 |
|---|---|
| 토글 상태 | ON / OFF |
| 버튼 상태 | 활성화 / 비활성화 |
| 색상 | 시나리오에 명시된 경우만 (한국어 색상명) |

## 12. TC 배치 순서

**원칙**:
1. 같은 L1(화면)을 한 블록으로 묶음 — 연속 배치
2. 블록 내에서 확정된 시나리오 순서를 따름 (기획서 § 순서)
3. Figma export(`figma_exports/{product}_{version}/`)가 있으면 UI 배치 순서 검증

**화면 영역 순서 (참고)**:
```
① 헤더 / 메뉴 → ② 검색·필터 영역 → ③ 액션 버튼 영역
→ ④ 본문 영역 (리스트·카드·테이블) → ⑤ 등록/수정 패널 → ⑥ Footer
```

**예외/에러 케이스 배치**:
- 예외 케이스를 별도 섹션으로 분리하지 않음
- 해당 UI 요소의 하위 조건으로 인라인 배치
- 같은 버튼의 정상/예외가 하나의 시나리오 흐름 안에 표현

---

## 13. 활용 흐름

1. **TC 작성**: JSON으로 로컬 저장
2. **검토**: review-tc 스킬로 형식·내용 검증
3. **시트 업로드**: gspread API 또는 Google Sheets MCP로 업로드 (옵션)
4. **자동화**: write-test-code에서 TC JSON 참조해 pytest 코드 생성
5. **버그 등록**: file-bug에서 Step·약어 ID 활용해 JIRA 등록

## 14. 안티패턴 (하지 말 것)

| 안티패턴 | 대안 |
|---|---|
| L2에 `권한`·`중복 환자 보호` 같은 카테고리·기능명 | L2는 영역, 카테고리는 note prefix |
| Step에 "로그인 후 진입" 같은 setup 동작 | 사전조건으로 분리 |
| Expected에 `- ` 불릿만 잔뜩 | 번호 매김 + 분해 |
| TCID 비우기 | `<도메인>-<L2약어>-NNN` 부여 |
| 사전조건에 기대결과 표현 옮기기 | 데이터 셋업·상태만 |
| 6단으로 잘게 쪼개기 | L1/L2/L3 3단 제한 |
| 한 행에 멀티 step + 멀티 expected | 행 분해 (Step 그룹 단위) |
| note에 케이스 유형 prefix | 액션 필요 항목만 |
