## 0. 한눈에 보기

```
TC = (TCID) + (어디서: L1·L2·L3) + (Given) + (When) + (Then) + (Priority) + (Note)
```

- 분류는 **"어디서"** 만 표현 (위치·영역·요소)
- 케이스 유형 (정상/권한/경계 등)은 분류가 아니라 **사전조건·note**로 표현
- Step과 Expected는 **행 단위 1:1** 매핑 — 멀티 step 시 행 분해 (한 행 = 한 액션 + 한 검증)
- **사전조건·Step은 `N. ` 번호 매김** (1줄이어도 `1. ` prefix). **기대결과는 한 행 1개라 번호 없음**
- 정책 근거는 note에 액션 필요 항목만 인용 (M/G/C 코드·기획 확인 필요 등)

## 1. 분류 기준 (L1 / L2 / L3)

### 1.1 패턴 — 패턴 A (화면 중심) 기본 권장

| 패턴 | 정의 | 권장 상황 |
|---|---|---|
| **패턴 A (화면 중심)** | L1=화면 / L2=UI 영역 / L3=UI 요소·동작 | **기본 권장** — UI 단위 운영에 자연스러움 |
| 패턴 B (기능 중심) | L1=기능 도메인 / L2=세부 기능 / L3=동작 | 화면 여러 곳에 걸친 기능 묶음이 더 중요할 때 |
| 패턴 C (하이브리드) | L1=화면 / L2=UI 영역 / L3=요소 — 케이스 | L3만 봐도 케이스 식별 필요한 운영 |

> 프로젝트 시작 시 한 패턴 선택. 이후 일관 적용. 변경 시 마이그레이션 비용 큼.

### 1.2 각 레벨 정의 (패턴 A 기준)

| 레벨 | 기준 | 어휘 가이드 |
|---|---|---|
| **L1 (대분류) — 화면** | 헤더 메뉴 단위 1:1 (또는 라우트 단위) | 메뉴명 그대로 (변형·축약 X) |
| **L2 (중분류) — UI 영역** | 기획서 § 섹션 단위 우선 (디자인 시안 박스 보조) | 명사형. 동작·상태·조건 포함 X |
| **L3 (소분류) — UI 요소·동작** | 영역 내 개별 위젯·입력·액션 | `[버튼명]` / `필드명` / `명사형 동작` |

### 1.3 L2 가상 영역 허용 (Cross-cutting·상태 케이스)

다음 두 종류는 일반 UI 영역이 아닌 **가상 L2**로 묶음.

**(1) Cross-cutting concerns** — 여러 영역을 가로지르는 횡단 관심사

| L2 이름 (예시) | 의미 | 약어 (예시) |
|---|---|---|
| 실시간 동기화 | SSE / 자동 갱신 / 동시성 검증 | `RTM` |
| 권한 검증 | 권한별 차단/허용 (여러 영역 흩어진 권한 케이스 모음) | `PRM` |
| 상태 전이 | 도메인 상태 전이 (여러 영역에서 트리거) | `STA` |
| 단축키 | 키보드 단축키 / 접근성 | `KEY` |

**(2) 상태·전역 케이스** — 특정 UI 영역에 안 속하는 검증

| L2 이름 | 의미 | 약어 (예시) |
|---|---|---|
| `(전체)` | 화면 전체 / 영역 비특정 / 레이아웃 | `ALL` |
| `(미구현)` | 영역으로 존재하지 않으나 기획상 있어야 할 항목 | `TBD` |

### 1.4 작성 규칙

1. **L1은 헤더 메뉴명 그대로** — 변형·축약 X
2. **L2는 기획서 § 제목 그대로** 사용 권장
3. **L2가 기획서·디자인 어디에도 없는 경우** — 가상 L2(§1.3) 사용
4. **L3는 명사형 / 명사형+액션** — `[버튼명]` 대괄호 일관
5. **분류는 "어디서"만 표현** — 동사·조건·케이스 유형은 분류에 넣지 X
6. **분류 컬럼 특수문자** — `[ ]` / `( )` 외 자제

## 2. 사전조건 (Given)

### 2.1 형식

```
1. <조건 1>
2. <조건 2>
   - <세부 1>
   - <세부 2>
3. <조건 3>
```

- **`N. ` 번호 매김 필수** — 한 줄이어도 `1. ` prefix
- 들여쓴 `- ` 서브 불릿으로 세부 상태 표현 (서브 불릿은 번호 매김 X)
- 들여쓰기 없는 줄 = 새 조건 / 들여쓴 줄 = 직전 조건의 세부

### 2.2 작성 카테고리

| 카테고리 | 예시 |
|---|---|
| **로그인·권한** | "{권한} 권한 계정 로그인" |
| **데이터 상태** | "환자 A가 미종료 내원 보유" / "예약 10건 등록된 상태" |
| **UI 상태** | "사이드바 펼쳐진 상태" / "접수 패널 열린 상태" |
| **백엔드/mock** | "{API}가 {조건} 실패하는 상황" |
| **시간** | "시작 시각으로부터 60분 이상 경과 상태" |

### 2.3 로그인·계정 권한 처리 (절충 방침)

**원칙**: 노이즈 줄이고 의미 있는 경우만 명시. 단 검증 대상 자체가 로그인·권한인 경우는 반드시 명시.

**(1) 로그인 상태**

| 경우 | 명시 여부 |
|---|---|
| 일반 TC (조회·필터·UI 노출·동작 등) | **명시 불필요** — 로그인 상태가 기본 전제 |
| **화면 진입 검증** (헤더·사이드바 메뉴 진입 액션) | **명시 필요** (`로그인 상태` 또는 `비로그인 상태`) |
| **로그인 테스트** 영역 | **필수 명시** + 계정 상세 |

**(2) 계정 권한**

| 경우 | 명시 여부 |
|---|---|
| **동작 권한 필요 TC** — 등록·수정·삭제·상태 변경·결제·드래그·일괄 처리 | **명시** |
| **권한 분기 검증** (PRM 가상 L2) | **명시** (차단되는 권한 또는 그룹) |
| 단순 조회·필터·UI 노출·진입·레이아웃 | **제거** (권한이 결과에 영향 없음) |

**(3) 로그인 테스트 영역 — 계정 정보 상세**

다음 항목을 사전조건에 명시:
- 로그인 ID / 비밀번호 입력 유무·정합성 (정상·오타·공란·SQL injection 시도 등)
- 계정 상태 (정상·잠금·만료·삭제·휴면)
- 세션 상태 (이미 로그인됨·세션 만료됨)
- 단말·브라우저 환경 (필요 시)

**(4) 권한 표기 형식**

- 단일 권한: `{권한명} 권한 계정 로그인`
- 허용 권한 그룹: `{권한1}/{권한2}/{권한3} 중 1 권한 로그인`
- 권한 차단 검증: 차단되는 권한만 명시

### 2.4 분해 행 사전조건 누적 규칙 (범용)

**기본 원칙**: Step 텍스트가 자기 충족적으로 작성되면 사전조건 누적 거의 불필요. 누적은 후속 step이 **직전 결과를 데이터·도메인 상태로 전제**하는 경우에만.

**판정 표**

| 직전 expected 종류 | 누적 여부 | 이유 |
|---|---|---|
| 일시 UI 상태 (창/팝업/드롭다운/메뉴/토스트) | **X** | Step에서 "X 창에서 ...Tap" 자체 표현 가능 |
| 단순 UI 노출 (목록·카드·뷰·라벨 노출) | **X** | 후속 step과 무관 |
| 부수 효과 (같은 액션의 추가 결과) | **X** | 이미 서브 불릿으로 흡수 |
| **영구 데이터 변경** (생성/수정/삭제/상태 전이) | **O** | 후속 step이 그 상태 전제 |
| **데이터 셋업 변화** (필터 적용·검색 결과) | **O** | 후속 step의 시작점 |

**자동 식별 키워드 사전 (범용)**

**일시 UI 키워드 (누적 X)**

| 카테고리 | 키워드 |
|---|---|
| 모달성 | 창, 팝업, 모달, 다이얼로그, 컨텍스트 메뉴, 선택 상자, 콤보박스 |
| 보조 UI | 드롭다운, 툴팁, 포퍼 |
| 토스트성 | 토스트, 스낵바 |
| 입력 보조 | 자동완성, 제안 목록 |

**영구 변경 동사 (누적 O)**

| 카테고리 | 동사 |
|---|---|
| 생성/삭제 | 등록됨, 추가됨, 생성됨, 삭제됨, 제거됨 |
| 수정 | 변경됨, 수정됨, 갱신됨, 업데이트됨 |
| 상태 전이 | 전이됨, 이동됨, 전환됨, 처리됨, 완료됨, 종료됨, 시작됨 |
| 활성화 | 활성화됨, 비활성화됨, 적용됨, 해제됨 |
| 수치 변화 | 증가됨, 감소됨, 차감됨, 적립됨, 누적됨 |
| 통신 | 저장됨, 발송됨, 수신됨, 전송됨 |

**판정 우선순위**

1. 직전 expected에 **일시 UI 키워드** 매칭 → **누적 X**
2. 직전 expected에 **영구 변경 동사** 매칭 → **누적 O**
3. 둘 다 매칭 X → **단순 UI 노출 판단** → **누적 X** (보수적 기본값)

**도메인 확장 가이드**

새 도메인·서비스 적용 시 키워드 사전 보강:
- 일시 UI: 해당 서비스의 모달/팝업 명칭
- 영구 변경: 도메인 특유의 상태 전이 동사

## 3. Step (When)

### 3.1 형식

단일 액션:
```
1. <액션>
```

멀티 액션:
```
1. <액션 1>
2. <액션 2>
3. <액션 3>
```

- **`N. ` 번호 매김 필수** — 한 줄이어도 `1. ` prefix
- Tap / Click / 입력 / 드래그 등 동작 명시
- 버튼·메뉴는 `[버튼명]` 대괄호 표기 일관

### 3.2 규칙

1. **한 Step = 한 사용자 액션** (마우스 1회, 키보드 1회, 드래그 1회 등)
2. **Step과 Expected는 행 단위 1:1 매핑** — 멀티 step은 행 분해(§4.2)로 표현
3. **준비 동작은 사전조건으로 빼기** — 시나리오 흐름상 검증 가치 있으면 Step에 포함 OK
4. **Step 텍스트로 자기 충족성 확보** — "X 창에서 [확인] Tap" 같이 시작 맥락 명시

## 4. 기대결과 (Then)

### 4.1 형식

```
<결과>
   - <세부 검증 1>
   - <세부 검증 2>
```

- **번호 매김 없이** 한 행에 한 결과만 기재 (분해 옵션 B 기준)
- 같은 결과의 부수 효과는 `- ` 서브 불릿
- DB·서버 상태 변화도 명시 (단 본문은 비즈니스 용어로)

### 4.2 분해 규칙 — Step 그룹 단위

분해는 **expected 줄 단위가 아니라 step 그룹 단위**.

**분해 알고리즘**

1. Step과 Expected 수 매칭 패턴:
   - Step N == Expected N → 1:1 매핑 (단계별 검증)
   - Step 1 + Expected N → 한 TC, expected들 서브 불릿
   - Step N + Expected 1 → 한 TC, 모든 step을 한 셀에 묶음 (번호 재부여)
   - Step M + Expected N (M<N) → 처음 M-1은 1:1, 나머지는 마지막 step에 묶임
2. 각 expected를 step 인덱스에 매핑 후 인접 같은 step 인덱스 = 한 그룹 (= 한 TC 행)
3. 다른 step 인덱스 = 별도 그룹 (= 별도 TC 행, 연쇄)
4. 한 그룹 안 멀티 expected는 첫 expected(메인) + 나머지 서브 불릿(`   - ...`)
5. **분해된 각 행의 Step 번호는 1부터 재부여** — 원본 부모 TC의 step 번호(2, 3, ...)가 그대로 들어가지 않도록. 각 행 안에서 Step은 항상 `1. `부터 시작

**연쇄 vs 비연쇄**

| 조건 | 판정 | 사전조건 처리 |
|---|---|---|
| 후속 행 Step ≠ 직전 행 Step | 연쇄 | §2.4 누적 규칙 적용 |
| 후속 행 Step == 직전 행 Step | 비연쇄 (같은 그룹에 묶임) | — |

### 4.3 별도 TC로 분리해야 하는 케이스

분해는 한 시나리오 안에서 step→expected 단위로 쪼개는 거고, **시나리오 자체가 다르면 별도 부모 TC로 분리**.

| 패턴 | 처리 |
|---|---|
| 동일 시작점, **다른 액션 조합** | 별도 부모 TC |
| 동일 액션, **다른 사전조건/데이터** | 별도 부모 TC |
| 동일 액션, **다른 권한** (결과 분기) | 별도 부모 TC (또는 PRM 가상 L2) |
| 동일 시작점, **순차 액션** (이어지는 흐름) | 한 부모 TC, 분해로 처리 |

**판단 질문**:
- 검증 의도가 같은가? → YES 분해 / NO 별도
- 다른 데이터 셋이 필요한가? → YES 별도
- 이 행만 단독 실행 가능한가? → YES 별도가 자연스러움

## 5. ID 명명 규칙

### 5.1 도메인 prefix (3자 약어)

프로젝트별 도메인 단위로 3자 약어 사전 정의. 예시 (DAY 프로젝트):

| 도메인 | prefix |
|---|---|
| 접수탭 | `RCP` |
| 시술탭 | `TRT` |
| 수납탭 | `PAY` |
| 마감탭 | `CLS` |
| 예약관리 | `RSV` |

> 신규 도메인 추가 시 약어 사전 갱신 후 적용.

### 5.2 번호 체계 — 약어 ID

**기본 형식**: `<도메인약어>-<L2약어>-<NNN>` (하이픈 구분, 3단)

- 예: `RSV-FLT-001` = 예약관리 / 좌측 필터 패널 / 1번째 TC
- 도메인 약어 (3자), L2 약어 (3자), 시퀀스 (3자리 zero-padded)

**분해 행 처리**:
- 한 기대결과 = 한 TCID = 한 독립 TC
- 분해된 각 행은 별개 unique TCID (sub-index `_01`, `_02` 사용 X)
- 시트에서 셀 병합 없이 각 행이 완전 독립
- 결과 트래킹·BTS ID도 행 단위 독립 기록

**규칙**:
- 신규 추가 시 해당 (도메인, L2) 마지막 번호 +1 (gap 없이)
- 새 L2 영역 등장 시 새 약어 정의 + 약어 사전에 추가
- 삭제 시 번호 재사용 X (이력 보존)
- 분류 변경 시 L2 약어가 바뀌면 새 ID로 갈음

### 5.3 L2 약어 사전 작성 가이드

도메인별로 L2 약어 사전을 프로젝트 시작 시 정의. 형식:

| 약어 | L2 영역 | 시나리오 § |
|---|---|---|
| `ENT` | 화면 진입 | §1 |
| `FLT` | 좌측 필터 패널 | §2 |
| ... | ... | ... |

**약어 작성 원칙**:
- 3자 고정 (일관성)
- 영문 알파벳 (URL·파일명·코드 호환)
- 의미 있는 약어 (단순 시퀀스 X)
- 가상 L2 약어: `ALL` / `TBD` / `RTM` / `PRM` / `STA` 등 권장

## 6. Note (비고) 컨벤션

### 6.1 비고에 입력하는 것 — **액션 필요 항목만**

비고는 **추가 작업·논의·확인이 필요한 항목**만 기재. 정상 케이스·일반 정책 인용은 비우기.

**비고에 들어가는 것:**
- **`M` 코드** — 미구현 (예: `M05 — 종료 상태 드롭다운 비활성화 미구현`)
- **`G` 코드** — 기획 부재·정의 부재 (예: `G09 — 초과 입금 정책 부재`)
- **`C` 코드** — 정책 충돌·기획서·정책서 불일치 (예: `C02 — 결제와 transition은 분리`)
- **자유 텍스트** — "기획 확인 필요", "운영 정책 검토 필요" 등 액션 항목

**비고에 넣지 않는 것:**
- 케이스 유형 분류 (정상/권한/경계/네거티브/실시간)
- 일반 정책 인용 (정상 동작 근거)
- 이슈 키 (`{PROJECT}-XX`) — 결과 트래킹 시 BTS ID 컬럼에 기록

### 6.2 형식

```
<코드> [/ <코드 추가>] — <짧은 설명>
```

복수 코드 가능: `M48 / C07 — 정책 = 확정, 코드 = 대기 (코드 수정 필요)`

### 6.3 정책 근거·시나리오 §·이슈 키는 어디에?

- **정책 근거** = 분류·사전조건·기획서 § 위치로 자연 추적 가능
- **시나리오 §** = TC ID의 L2 약어로 매핑됨
- **이슈 키** = 실행 결과 시 BTS ID 컬럼에 기록

## 7. Priority 기준

| 등급 | 적용 케이스 |
|---|---|
| **P1** | 핵심 흐름 (smoke). 깨지면 출시 막힘. 권한·인증·결제·중복 차단·핵심 상태 전이 |
| **P2** | 주요 기능. 경계·네거티브 중 영향 큰 것. 실시간 동기화. 정상 흐름의 변형 |
| **P3** | 보조 기능·UI 디테일. 색상·정렬·미세 인터랙션. 미구현 항목 |

**기본 배분 가이드**: P1 ~25% / P2 ~45% / P3 ~30%

## 8. 안티패턴 (하지 말 것)

| 안티패턴 | 왜 안 좋은가 | 대안 |
|---|---|---|
| L2에 `권한` / `중복 환자 보호` 같은 카테고리·기능명 | 영역이 아님 — 분류 일관성 깨짐 | L2는 영역, 카테고리는 note prefix |
| Step에 "로그인 후 진입" 같은 setup 동작 | 매번 반복되어 핵심 흐름 흐림 | 사전조건으로 분리 |
| Expected에 `- ` 불릿만 잔뜩 | Step과의 매핑 불명 | 번호 매김 + 분해 |
| TCID 비우기 (행 위치만 사용) | 호명·참조·grep 불가능 | `<도메인약어>-<L2약어>-NNN` 부여 |
| note 비우기 | 액션 필요 항목 추적 불가 | M/G/C 코드·자유 텍스트 |
| 6단으로 잘게 쪼개기 | 컬럼 dead율 ↑, AI 추론 부담 ↑ | L1/L2/L3 3단으로 제한 |
| 모든 권한별로 TC 분리 | 행 폭발 | 옵션 ② 결과 동일 권한 그룹핑 |
| 사전조건에 기대결과 옮기기 | 의미 혼동 | 데이터 셋업·상태만 |

## 9. 좋은 예 vs 나쁜 예

### 9.1 권한 케이스

**나쁜 예**
```
L1=접수탭 / L2=권한 / L3=체크인 권한
사전조건: (없음)
Step:    의사로 로그인, 체크인 시도
기대:    권한 없음
note:    (없음)
```
문제: L2가 카테고리 / Step에 로그인 setup 포함 / note 없음.

**좋은 예**
```
L1=접수탭 / L2=접수 패널 / L3=[체크인] 버튼
사전조건: 1. 의사 권한 계정 로그인
          2. 접수탭 진입 상태
Step:    1. 신규 환자 체크인 시도
기대:    "권한이 없습니다" 알림 노출됨
note:    (정상 권한 분기라 빈칸)
```

### 9.2 멀티 step 분해

**나쁜 예** (한 행에 멀티 step + 멀티 expected)
```
Step:     드래그 / 담당자 선택 / 확인 Tap
Expected: 드래그 효과 / 모달 열림 / 카드 이동
```
문제: 어느 step에서 fail인지 추적 불가.

**좋은 예** (행 분해 — Step 그룹 단위)
```
RSV-CAL-001  Step="카드 드래그"           Expected="드래그 효과 노출"
RSV-CAL-002  Step="담당자 모달에서 선택"  Expected="카드 이동"
```

### 9.3 사전조건 누적

**연쇄 케이스 (누적 O — 영구 변경)**
```
TC-A: Step "[등록] 버튼 Tap" / 기대 "예약이 등록됨"  ← 영구 변경
TC-B: 사전조건에 "예약이 등록됨" 누적 → Step "등록된 행 클릭"
```

**비연쇄 케이스 (누적 X — 일시 UI)**
```
TC-A: Step "드롭다운 Tap" / 기대 "옵션 목록 노출됨"  ← 일시 UI
TC-B: 사전조건 원본 유지 → Step "드롭다운에서 '이행' 선택"
```

## 10. 시트 ↔ JSON 매핑

### 10.1 컬럼 매핑

| 시트 컬럼 | JSON 필드 |
|---|---|
| TCID | `id` |
| 대분류 | `category_l1` |
| 중분류 | `category_l2` |
| 소분류 | `category_l3` |
| 사전조건 | `precondition` |
| Step | `step` |
| 기대결과 | `expected` |
| Priority | `priority` |
| Result | `test_result.status` |
| 담당자 | `test_result.assignee` |
| Date | `test_result.executed_at` |
| BTS ID | `test_result.bts_id` |
| 비고 | `note` |

### 10.2 source of truth

- **JSON = source of truth** (git diff·PR 리뷰·자동화 파이프라인용)
- **시트 = read-only 미러** (PM/QA 회의·수동 결과 입력용)
- 시트에서 직접 수정한 결과(P/F/B + 담당자/Date/BTS)는 별도 도구로 JSON에 역동기화

## 11. 스킬 연계

이 규칙 문서는 다음 스킬들의 reference로 사용됨:

| 스킬 | 이 규칙 어떻게 쓰나 |
|---|---|
| `write-scenario` | 시나리오 § 구조 ↔ L2 매핑 의식해서 작성 |
| `write-tc` | 시나리오 → TC 변환 시 본 규칙 강제. precondition/step/expected 분리·번호 매김·분해 자동 적용 |
| `review-tc` | TC 검증 시 체크리스트로 사용. L2 카테고리 검출, Step↔Expected 매핑, note prefix 누락 등 |
| `write-test-code` | Step N → assert N 매핑으로 pytest 코드 결정론적 생성 |
| `file-bug` | Step 컬럼 그대로 재현 절차로 복사 + 약어 ID로 컴포넌트 자동 매핑 |
| `review-bug` | 같은 규칙 기준으로 버그 리포트 검증 |
| `check-format` | 형식 규칙 자동 검증 (번호 매김, 분류 정합성, 약어 ID 형식) |
| `review-spec` / `review-qa` | 분류 기준·시나리오 § 매핑 의식 |
